## 1. 背景与目标

### 1.1 痛点

文件服务器长期使用后，普遍面临三个问题：

- **重复文件堆积**：同一份资料散落多个目录，占用大量空间
- **整理工具缺位**：fdupes、rmlint 只能查重，批量整理还要靠命令行
- **没有友好的 web 界面**：每次操作都得 SSH 到服务器，对非技术同事不友好

### 1.2 目标

做一个**单机 web 工具**，解决上述三个问题：

1. 接收服务器导出的 md5_list.txt（含 MD5 + 完整路径），导入到本地 SQLite
2. 浏览器端查重、整理、预览
3. 系统**不直接操作文件服务器**——生成 shell 命令让人在服务器执行

### 1.3 非目标（明确不做）

- ❌ 不直连文件服务器（避免 SSH、sudo、远程协议复杂度）
- ❌ 不做实时同步（人工确认即可，简单可控）
- ❌ 不做用户认证（1-2 人内网，零安全需求）
- ❌ 不做文件上传/下载（只管元数据，不管内容）
- ❌ 不做文件预览/转码（▶️ 播放是直接调 mac 系统播放器）

### 1.4 核心设计哲学

> **"系统是只读元数据库 + 命令生成器，操作权永远在用户手里。"**

- 系统从不直接 rm/mv 文件
- 系统只生成命令、复制、等待用户执行、回点确认
- 审计日志永久保留，可回溯
- 这套设计绕开了"自动 rm 误删"的恐惧，也避开了远程协议的实现成本

## 2. 用户与场景

### 2.1 目标用户

- 内部 1-2 人
- 局域网内访问 web（http://本机IP:5000）
- 同时能 SSH 到文件服务器执行命令
- 有 Linux 命令行基础

### 2.2 典型场景

#### 场景 A：清理重复文件

1. 在文件服务器生成 md5_list.txt：
   ```bash
   find /volume1 -type f -exec md5sum {} \; > md5_list.txt
   ```
2. 浏览器打开 web → 重复文件页 → 顶部"📥 导入"按钮 → 上传 md5_list.txt
3. 看到 N 组 MD5 重复
4. 在每组里勾选要删的（系统强制每组至少留 1 个）
5. 可点 ▶️ 播放查看文件内容
6. 点"生成删除命令" → 复制 `rm /path1 /path2 ...`
7. 粘到服务器终端运行
8. 回 web 点"已执行" → 数据库同步

#### 场景 B：批量整理文件

1. 打开"资源管理器"
2. 左侧选目录，右侧文件列表支持搜索
3. 每行 3 个按钮：▶️ 播放 / 🗑 删除 / 📁 移动
4. 勾选文件 → 点"移动到..." → 弹窗选目标目录（必须在库中已有）
5. 生成 `mv -i /src /dst` → 复制 → 服务器执行
6. 回 web 点"已执行" → 数据库更新路径

#### 场景 C：边整理边预览

- 文件行点 ▶️ 播放按钮
- 系统拼 `smb://<server>/<path>` → `open` 命令 → mac 默认播放器启动
- 不影响数据，只用来看一眼文件再决定怎么处理

### 2.3 工作流总览

```
[文件服务器]                      [Mac 本机]
                                  
md5_list.txt  ──上传──>      [Flask Web]  ──importer──>  SQLite
                                 │
                            [Flask Web]
                            ├─ 重复文件 (rm + ▶️)
                            └─ 资源管理器 (mv + rm + ▶️)
                                 │
                            生成 shell 命令
                                 │
                  <用户 copy 到服务器>
                                 │
                            服务器执行
                                 │
                  <用户回 web 点"已执行">
                                 │
                            DB 同步
                                 │
                            ↻ 两边平衡
```

## 3. 功能详述

### 3.1 页面 1：重复文件（Tab 结构）

**页面 1 改为 Tab 切换**：

```
[页面 1]
┌──────────────────────────────────────┐
│  [ MD5 重复 ]  [ 同名 ]              │  ← tab 切换
├──────────────────────────────────────┤
│  [📥 导入]  [🔍 扫描同名]            │  ← 按钮随 tab 切换
│                                      │
│  Tab 1: md5 重复组列表...            │
│  Tab 2: 同名组列表 + 每行 [忽略]     │
└──────────────────────────────────────┘
```

#### Tab 1: MD5 重复

- 顶部"📥 导入 md5_list.txt"按钮
- 按 MD5 分组（>1 个为重复）
- 组内按 **文件名升序**（组间排序待 UI 阶段定）
- 每组展示：MD5 前 12 位、文件数、总大小
- 每行：
  - 勾选框（批量删除用）
  - 文件名 / 路径 / 大小
  - **▶️ 播放**按钮（单文件预览，方便决定删哪个）
  - **"只留这个"**按钮（一键勾选组内其他所有）

**操作流程**：
1. 批量勾选要删除的文件
2. 强制校验：**每组至少留 1 个**（前端阻止"全选"）
3. 点"生成删除命令" → 后端返回 `rm /path1 /path2 ...`（无 -v，最简形式，**1 文件 1 行**）
4. 弹窗显示命令，一键复制
5. 用户去服务器执行
6. 回 web 点"已执行" → 后端 `audit_log` + `DELETE` → 列表自动刷新

**边界**：
| 场景 | 行为 |
|------|------|
| 选 0 个 | 按钮 disabled |
| 全选一组 | 前端 alert 阻止 |
| 文件已被删（库里不存在） | confirm 时返回 404 |
| 该组原本 3 个，删了 2 个 | 该组从页面 1 消失，剩余 1 个文件 md5 唯一 |
| 有 pending task | "生成命令"按钮 disabled（全局 task 锁定） |

**导入按钮**：
- 位置：状态条右侧
- 点击：弹出模态框（文件选择 + 上传按钮 + 进度显示）
- 上传成功：刷新重复组列表
- 统计显示：新增 X / 更新 Y / 跳过 (.DS_Store) Z

#### Tab 2: 同名（Phase 2）

- 顶部"🔍 扫描同名"按钮
- 点击后扫描数据库，列出"文件名相同但路径不同"的组

**组卡片设计**：

```
[同名组卡片]
┌────────────────────────────────────┐
│ 文件名: 1.mp4 (3 个)       [忽略]  │  ← 组级忽略按钮
├────────────────────────────────────┤
│ ☐ /dir1/1.mp4  abc...              │  ← 勾选 + 文件信息
│ ☐ /dir2/1.mp4  def...              │
│ ☐ /dir3/1.mp4  ghi...              │
│ 每行: [▶️] [🗑]                     │  ← 单文件操作
└────────────────────────────────────┘
```

**扫描过滤规则**：
```sql
-- 找 file_name 出现 ≥ 2 次的组
-- 组内全部 ignored_at 不为 NULL → 跳过整个组
-- 组内至少 1 条 ignored_at 为 NULL → 整组列出（包括已忽略的）
```

**"忽略"按钮（组级，强制全选）**：
- 必须在**组卡片头部**
- **必须全选组内所有**才能点
- 没全选 → 报错"有相同文件没有确认，请全部选中后再次点击忽略"
- 全选 → UPDATE 所有行的 `ignored_at = NOW`

**每行操作按钮**：
- **▶️ 播放**：直接调 mac 默认播放器
- **🗑 删除**：加入暂存区（走 Phase 1 暂存区 + task 流程）

### 3.2 页面 2：资源管理器

#### 布局

```
┌──────────────┬──────────────────────────────────────────┐
│ 目录         │ 当前目录名     [搜索框]                  │
│ ─────        │ ────────────                             │
│ 📁 全部 (1k) │ ☐ 文件名  路径  大小  MD5  操作         │
│ 📁 dir1 (50) │ ☐ a.mp4   ...   100MB  abc  ▶️🗑📁      │
│ 📁 dir2 (30) │ ☐ b.jpg   ...   2MB    def  ▶️🗑📁      │
│ ...          │ ...                                     │
│              │ [已选 N 个] [移动] [删除] [清空]        │
└──────────────┴──────────────────────────────────────────┘
```

#### 每行 3 个按钮

| 图标 | 操作 | 行为 |
|------|------|------|
| ▶️ | 播放 | 直接调 mac 默认播放器（无需选中） |
| 🗑 | 删除 | 自动勾选该文件 + 生成 rm 命令（批量入口） |
| 📁 | 移动 | 自动勾选该文件 + 打开移动弹窗（批量入口） |

> 🗑 和 📁 的"自动勾选"是优化体验——单文件操作不用先去勾。

#### 目录树

- 顶部"📁 全部"显示所有文件数
- 每个目录显示文件数
- 搜索框实时过滤路径

#### 文件列表

- 默认按 `parent_dir, file_name` 排序
- 列：勾选 / 文件名 / 路径 / 大小 / MD5 前 12 位 / 操作
- 顶部搜索框：模糊匹配 `file_name` + `file_path`
- 切换目录时清空选择
- 库锁定时每行加 🔒 标记

#### 操作 - 移动

1. 勾选文件（可多选 + 📁 单文件入口）
2. 点"移动到..." → 弹窗
3. 弹窗内：搜索 + 树形选择目标目录
4. 校验：目标目录必须在库中有 file 记录
5. 点"生成 mv 命令" → 后端返回：
   ```
   mv -i '/volume1/a.mp4' '/volume1/dst/a.mp4' && \
   mv -i '/volume1/b.jpg' '/volume1/dst/b.jpg'
   ```
6. 复制 → 执行 → 点"已执行" → 后端 `audit_log` + `UPDATE file_path`

#### 操作 - 删除

1. 勾选文件（可多选 + 🗑 单文件入口）
2. 点"删除" → 生成 `rm /path1 /path2 ...` → 弹窗显示
3. 复制 → 执行 → 弹窗内"已执行"按钮 → `audit_log` + `DELETE`

#### 操作 - 播放

1. 点 ▶️ 按钮
2. 后端拼 `smb://<server-ip>/<相对路径>`（剥第一级）
3. 后端 `subprocess.Popen(['open', url])`
4. mac 系统调用默认播放器

### 3.3 锁定机制

系统有两层锁定机制。

#### 锁定 1: 库级锁定（MD5 重复）

> **库中存在任意 md5 出现 ≥ 2 次时，整个页面 2 进入只读模式。**
> **必须先在页面 1 把所有重复组消化完（每组至少留 1 个），库级才解锁。**

**前端表现**：
- 列表顶部状态条：⚠️ 库中还有 N 组重复未处理 → [去处理]
- 每个文件行加 🔒 标记 + hover 提示
- 复选框、按钮全部 disabled
- 库级解锁后，状态条消失，UI 恢复

**API 支持**：
- `/api/tree` 返回 `library_locked: bool` + `locked_group_count: int` + `locked_md5_set: [...]`
- 前端用 `locked_md5_set.has(file.md5)` 判断单文件 `is_locked`
- `/api/files` 每行加 `is_locked: bool`
- 写操作 API（generate-rm/mv、confirm-rm/mv）检查锁定，发现锁定状态返回 **403**

**性能**：
- 锁定判定：单次 SQL `SELECT md5 FROM files GROUP BY md5 HAVING COUNT(*) > 1`
- md5 已建索引，毫秒级
- **不缓存**，每次请求实时算（保证一致性）

#### 锁定 2: Task 全局锁定（pending task）

> **任一 task 处于 `pending` 状态时，所有页面的"操作"按钮都 disabled。**
> **必须点击"已执行"使 task 变为 `executed`，全局才解锁。**

**前端表现**：
- 顶部状态条：⚠️ 有 pending task，请到 [Task 页面] 处理
- 页面 1（tab 1 + tab 2）的"生成命令"按钮 disabled
- 页面 2 的"移动 / 删除"按钮 disabled
- ▶️ 播放（不修改数据）不受影响

**API 支持**：
- `/api/tree`、`/api/duplicates` 等查询 API 返回 `pending_task_exists: bool` + `pending_task_count: int`
- 写操作 API（generate-rm/mv、confirm-rm/mv）检查 pending task，发现返回 **403**

**为什么全局锁定**：
- 保证一个 task 对应一次服务器执行批次，状态清晰
- 避免多个 task 并行导致 DB 状态错乱
- 强制用户按"暂存 → 生成 → 执行 → 确认"的节奏走

#### 两层锁定独立

- 库级锁定（MD5 重复）和 Task 锁定独立判断
- 状态可以叠加：库锁定 + pending task → 全只读
- 任一锁定解除不解除另一个

### 3.4 ▶️ 播放功能

#### 配置

`config.json`（项目根目录）：
```json
{ "smb_server_ip": "192.168.1.100" }
```

只有一个字段。share 名**不用单独配**——自动从路径第一级推断。

#### 路径转换（不对原始路径做任何调整）

数据库存的就是服务器导出的原始路径（如 `/volume1/GV/a.mp4`），**不做任何调整**。

- 原始路径（库内）：`/volume1/GV/a.mp4`
- 剥掉第一级 `/volume1` → `/GV/a.mp4`
- 拼：`smb://192.168.1.100/GV/a.mp4`
- 调用 `subprocess.Popen(['open', url])` → mac 默认应用

**为什么不做 mac mount 路径转换**：
- 原始数据来自服务器，保留原样最准确
- mac mount 路径（`/Volumes/Volume1/...`）会随 mount 变化，不入库
- 播放时直接对原始路径"剥第一级"，逻辑简单

#### 边界

| 场景 | 行为 |
|------|------|
| 路径不以 `/<share>/` 开头（实际不会出现） | 按钮 disabled |
| 配置文件缺失或 IP 为 `0.0.0.0` | 全局按钮 disabled，hover 提示"先配置 SMB" |
| 协议不是 `smb://` | 后端拒绝（防注入） |
| 首次启动 | 自动生成 `config.json` 模板（占位 IP）+ 控制台提示 |

### 3.5 导入时过滤 .DS_Store

导入 md5_list.txt 时，**自动跳过 `.DS_Store` 文件**：

- mac 系统在访问过的目录会生成 `.DS_Store` 元数据
- 这些文件没有用户价值，会污染重复检测
- 解析时检查路径是否包含 `/.DS_Store` 或以 `.DS_Store` 结尾
- 统计时计入"跳过"计数

## 4. 数据模型

### 4.1 files 表

```sql
CREATE TABLE files (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    file_path TEXT NOT NULL UNIQUE,
    file_name TEXT NOT NULL,
    parent_dir TEXT NOT NULL,
    md5 TEXT NOT NULL,
    file_size INTEGER,
    ignored_at TIMESTAMP,                       -- Phase 2 同名组"已忽略"标记
    imported_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_md5 ON files(md5);
CREATE INDEX idx_parent_dir ON files(parent_dir);
CREATE INDEX idx_file_name ON files(file_name);
```

**字段设计要点**：
- `file_path` UNIQUE：保证同一路径只一条记录（重导入不重复）
- `parent_dir`：冗余存储，让"目录树"和"目标选择"不用每次 split 路径
- `file_name`：冗余存储，搜索 / 排序时不用 split
- `ignored_at`：Phase 2 同名组的"已忽略"标记（NULL = 未忽略）
- 索引：`md5` 查重、`parent_dir` 目录浏览、`file_name` 搜索

### 4.2 audit_log 表

```sql
CREATE TABLE audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    file_path TEXT NOT NULL,
    md5 TEXT,
    action TEXT NOT NULL,         -- 'deleted' | 'moved'
    new_path TEXT,                -- 仅 moved 使用
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**永久保留**：所有 rm / mv 操作都记一行，可追溯"哪个文件何时被删/移到哪"。

### 4.3 操作语义

| 操作 | DB 行为 |
|------|---------|
| 删除 N 个 | audit_log + N 条 'deleted' + DELETE N 行 |
| 移动 N 个 | audit_log + N 条 'moved' (old_path, new_path) + UPDATE N 行的 file_path + parent_dir |
| 重新导入 md5_list.txt | ON CONFLICT(file_path) DO UPDATE（路径相同则更新元数据） |

## 5. API 设计

### 5.1 页面路由

| 路径 | 方法 | 作用 |
|------|------|------|
| `/` | GET | 首页（导航卡片 + 工作流说明） |
| `/duplicates` | GET | 重复文件页（顶部含"导入"按钮） |
| `/explorer` | GET | 资源管理器页 |

### 5.2 JSON API

| 路径 | 方法 | 作用 |
|------|------|------|
| `/api/duplicates` | GET | 列出所有 md5 重复组 |
| `/api/same-name` | GET | 列出所有同名不同 MD5 组（Phase 2） |
| `/api/ignore-same-name` | POST | 标记同名组为已忽略（Phase 2） |
| `/api/generate-rm` | POST | ids → rm 命令 |
| `/api/confirm-rm` | POST | ids → audit_log + DELETE |
| `/api/tree` | GET | parent_dir 树 + library_locked + locked_md5_set + pending_task |
| `/api/files` | GET | 列出文件（parent_dir + q 过滤） |
| `/api/generate-mv` | POST | ids + target_dir → mv 命令 |
| `/api/confirm-mv` | POST | ids + target_dir → audit_log + UPDATE |
| `/api/play` | POST | file_id → subprocess open smb:// |
| `/api/config-status` | GET | `{smb_configured: bool}` |
| `/api/import` | POST | 上传 md5_list.txt，返回导入统计 |
| `/api/tasks` | GET | 列出所有 task（pending + executed） |
| `/api/task-detail` | GET | 单个 task 的命令详情 |

### 5.3 关键 API 详述

#### POST /api/import

```json
// 请求：multipart/form-data
// file: <md5_list.txt>

// 响应
{
  "ok": true,
  "inserted": 12345,         // 新增
  "updated": 678,            // 路径相同但 md5 变了（更新）
  "skipped_ds_store": 23,    // .DS_Store 跳过
  "errors": 0,               // 解析错误行数
  "total_lines": 13046,      // 总行数
  "elapsed_ms": 1234
}
```

- 单文件上传上限 500MB（Flask `MAX_CONTENT_LENGTH`）
- 事务批量 INSERT（每批 1000 行）
- 导入完成自动算重复组
- 大文件按行流式解析，内存友好

#### POST /api/generate-rm

```json
// 请求
{ "ids": [1, 5, 12] }

// 响应（成功）
{
  "command": "rm '/volume1/GV/a.mp4' '/volume1/GV/b.mp4'",
  "count": 2,
  "paths": ["/volume1/GV/a.mp4", "/volume1/GV/b.mp4"]
}

// 响应（403 锁定）
{ "error": "包含 3 个处于 md5 重复锁定的文件，请先到重复文件页处理" }
```

- 路径用 `shlex.quote` 防空格
- 锁定检查：任一 id 的 md5 在 `locked_md5_set` 中 → 403

#### POST /api/confirm-rm

```json
// 请求
{ "ids": [1, 5, 12] }

// 响应
{
  "deleted": 2,
  "library_locked": true,
  "locked_group_count": 3
}
```

#### POST /api/play

```json
// 请求
{ "file_id": 5 }

// 响应（成功）
{ "ok": true, "smb_url": "smb://192.168.1.100/GV/a.mp4" }

404 - file not found
400 - 路径不以 / 开头
```

## 7. 配置

### 7.1 config.json

```json
{
  "smb_server_ip": "192.168.1.100"

```

- **每次请求读 config**，不缓存（修改即时生效）

- 首次启动自动生成模板
- 库内**只存原始路径**（如 `/volume1/...`），不存 mac mount 路径、不做任何转换

### 7.2 数据库

- SQLite WAL 模式
- 路径：`./data/files.db`
- 容量建议：单库 < 几 GB（再大建议按目录 hash 分库）
- 性能：百万级文件无压力，几千万也行

### 7.3 md5_list.txt 格式

**生成方式**（在文件服务器上）：
```bash
find /volume1 -type f -exec md5sum {} \; > md5_list.txt
```

**文件格式**（每行）：
```
<32位小写MD5>  <绝对路径>
```

示例：
```
178e097c50395b96bdf286f1d8f88a44  /volume1/GV/-008-压缩包无密码 qq 2669901936/zb13-QQ2669901936.mp4
ceba4e9f1bb1fced2e0e702a006aeeab  /volume1/GV/011710/.DS_Store
```

**导入时的处理**：
- ✅ 标准格式直接解析：`split(' ', 1)` 取 MD5 和路径
- ✅ 路径包含空格也能正确处理
- ✅ 中文路径支持（UTF-8）
- ⏭️ **自动跳过 `.DS_Store`**（mac 系统元数据文件，不入库）
- 📊 导入完成后实时算重复组（一次 GROUP BY）

## 9. 安全与限制

### 已知风险（用户接受）

- 1-2 人内网使用，**无认证**
- API 任意调用，攻击者可触发任意 `open smb://` URL
- 接受理由：内网小工具，安全成本不划算

### 内置防护

- `open` 命令的 URL 只允许 `smb://` 协议（防注入 `file://`、任意 shell）
- 操作前必须人工复制命令、SSH、执行、回到 web 确认
- 写操作检查库锁定状态

### 容量边界

- SQLite 单库 < 几 GB（> 几 GB 写入性能下降）
- 单次导入 md5_list.txt：500MB 上限（Flask MAX_CONTENT_LENGTH）
- 锁定判定查询：md5 索引存在，毫秒级

## 12. Phase 2 功能详述

### 12.1 增量导入

#### 背景

第一次导入后，文件服务器持续会有新文件。用户要识别"哪些是新文件"并入库。

#### 流程

1. **服务器侧**：
   - 用户把新文件放到 `_inbox/` 这种特殊文件夹
   - 跑 `find /volume1/_inbox -type f -exec md5sum {} \; > new_files.txt`
2. **Web 侧**：
   - 重复文件页 → 顶部"📥 导入" → 上传 `new_files.txt`
3. **入库逻辑**：
   - 解析上传文件
   - **过滤**：和库里**已有 MD5 对比**，移除相同 MD5 的（这些是"已经存在的"，不需要再入库）
   - 剩下的（**纯新 MD5**）批量 INSERT
   - **入库后跑 GROUP BY 扫全部重复组**（包括新插入的）
4. **后续处理**：
   - 如果有新产生的 MD5 重复 → 页面 1 处理（走 Phase 1 流程）
   - 页面 2 把新文件从 `_inbox/` 移到合适位置（走 Phase 1 暂存区 + task 流程）

#### 关键规则

- **库里有但新一批里没有的旧文件 → 不主动删**
- 删除必须由用户**主动点击**删除按钮才会生成 rm 命令
- 新文件之间**可能产生重复**（比如新上传 100 个文件中有 2 个 MD5 相同）→ 入库后**必须**扫重复组
- `_inbox/` 路径 web 不特殊处理，用户自己在页面 2 浏览

#### API 改动

- `/api/import` 增加 `mode` 参数：`first`（Phase 1 全量）/ `incremental`（Phase 2 增量）
- 增量导入响应增加 `filtered_by_existing_md5: int`（被 MD5 过滤的数量）

### 12.2 同名不同 MD5 检测

#### 背景

不同目录下可能有同名文件但 MD5 不同。这不是技术重复（MD5 不同），但可能是：
- 用户上传了同一份文件但版本不同（压缩包 v1 / v2）
- 用户在不同目录复制了不同文件
- **不一定是问题，但需要让用户看一眼判断**

#### 流程

1. 页面 1 Tab 2 → 顶部"🔍 扫描同名"按钮
2. 扫描 `file_name` 出现 ≥ 2 次的组
3. 过滤规则：
   - 组内所有记录的 `ignored_at` 都不为 NULL → **跳过整个组**（不再列出）
   - 组内至少 1 条记录的 `ignored_at` 为 NULL → **整组列出**（包括已忽略的）
4. 用户操作：
   - **组级"忽略"**（强制全选组内所有）：
     - 没全选 → 报错"有相同文件没有确认，请全部选中后再次点击忽略"
     - 全选 → UPDATE 所有行的 `ignored_at = NOW`
   - **单文件"删除"** → 加入暂存区，走 Phase 1 流程

#### Schema 改动

```sql
ALTER TABLE files ADD COLUMN ignored_at TIMESTAMP;
-- NULL = 未忽略；时间戳 = 忽略时间
```

#### API

- `GET /api/same-name`：返回过滤后的同名组
- `POST /api/ignore-same-name`：`{file_ids: [1, 5, 12]}` → UPDATE 所有行的 `ignored_at`

### 12.3 Task 系统

#### 数据模型

`tasks` 表：
```sql
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    status TEXT NOT NULL,             -- 'pending' | 'executed' | 'cancelled'
    command_type TEXT NOT NULL,       -- 'rm' | 'mv'
    commands TEXT NOT NULL,           -- 多条命令的 JSON 数组
    file_count INTEGER NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    executed_at TIMESTAMP
);
```

`staging` 表（暂存区）：
```sql
CREATE TABLE staging (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    file_ids TEXT NOT NULL,           -- JSON 数组
    target_dir TEXT,                  -- mv 用
    action TEXT NOT NULL,             -- 'rm' | 'mv'
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 流程

1. **加入暂存区**（多次累积）：
   - 页面 1 tab 1：勾选要删的文件 → 点"加入删除暂存"
   - 页面 1 tab 2：单文件点 🗑 → 加入删除暂存
   - 页面 2：勾选要移动的文件 → 选目标 → 加入移动暂存
2. **生成 task**（一次性）：
   - 暂存区点"生成命令" → 创建 1 个 task（pending）
   - 命令弹窗显示（1 文件 1 行 rm，或 mv 命令链）
3. **执行 + 确认**：
   - 复制命令到服务器执行
   - 回 task 页面点"已执行" → DB 同步 → task executed
4. **全局锁定**（pending 期间）：
   - 所有"操作"按钮 disabled
   - 顶部状态条：⚠️ 有 pending task
5. **取消**：pending task 可删除（cancel 状态）

#### 全局锁定规则

- 任一 task pending → 所有页面"操作"按钮 disabled
- 库级锁定和 task 锁定**独立判断**（可叠加）
- 浏览（看列表、看 ▶️ 播放）不受影响
